# Даалгаврын нийцлийн үнэлгээ

Огноо: 2026-07-06

Шалгасан файл: `анхны даалгавар/Даалгавар5.14_1.pdf`

Харьцуулсан файл: `төслийн concept/хөгжүүлэлт эхлэх судалгаа.md`

## 1. PDF даалгаврын товч агуулга

Даалгаврын үндсэн сэдэв:

**Системийн бүртгэл хийх**

Хэрэглэгчийн шаардлага:

- Хэрэглэгч нэвтрэх.
- Хэрэглэгчийн нэр болон нууц үг хадгалах.
- Системийн бүртгэл хийх.
- Системийн мэдээллийг жагсаалтаар харах, хайлт хийх.
- Мэдээллийн аюулгүй байдлын стандартуудыг шалгах.

Системийн бүртгэлийн заавал байх талбарууд:

- Системийн нэр.
- Төрөл: `Карт`, `Коре`, `Дотоод`, `Дижитал`.
- Үнэлгээ/төг.
- Тайлбар.
- Холбоотой системүүд.
- Хөгжүүлэгч.
- Хугацаа.
- Ашиглагдаж байгаа эсэх.

PDF дээрх эцсийн хугацаа: **2024-05-19**. Одоогийн огноо **2026-07-06** тул энэ нь өнгөрсөн хугацаатай reference даалгавар гэж үзнэ.

## 2. Нийцлийн дүгнэлт

Одоогийн `хөгжүүлэлт эхлэх судалгаа.md` даалгавартай **сайн нийцэж байна**, гэхдээ scope нь анхны даалгавраас нэлээд өргөн болсон.

Дүгнэлт:

- Даалгаврын үндсэн шаардлага бүр судалгаанд туссан.
- Enterprise Service Catalog санаа нь “Системийн бүртгэл хийх” даалгаврыг илүү мэргэжлийн болгож өргөтгөж байна.
- Гэхдээ эхний хөгжүүлэлтэд Gateway, Eureka, Config Server, олон microservice бүгдийг шууд хийх нь scope хэтэрнэ.
- Даалгаварт хамгийн түрүүнд хэрэгтэй зүйл бол: login, system CRUD, exact required fields, security standard check, list/search.

Товч үнэлгээ:

| Үзүүлэлт | Үнэлгээ |
| --- | --- |
| Даалгавартай нийцэл | Өндөр |
| Enterprise өргөтгөлийн үнэ цэнэ | Өндөр |
| Шууд хөгжүүлэлт эхлэхэд бэлэн байдал | Дунд-сайн |
| Scope хэтрэх эрсдэл | Дунд-өндөр |
| MVP болгож багасгах шаардлага | Заавал |

## 3. Шаардлага бүрийн mapping

| PDF шаардлага | Судалгаанд байгаа эсэх | Тайлбар | Хийх засвар |
| --- | --- | --- | --- |
| Хэрэглэгч нэвтрэх | Байгаа | Auth Service, JWT, role-based access гэж туссан | MVP-д username/password + BCrypt + JWT хийх |
| Хэрэглэгчийн нэр, нууц үг хадгалах | Хагас | `users` table байгаа | Нууц үгийг plain text биш `password_hash` болгон хадгалах |
| Системийн бүртгэл | Байгаа | Service Catalog CRUD гэж туссан | PDF-ийн exact талбаруудыг form/table/db-д заавал нэмэх |
| Системийн нэр | Байгаа | `services.name` | Шууд ашиглана |
| Төрөл: Карт, Коре, Дотоод, Дижитал | Хагас | `type` байгаа ч enum тодорхойгүй | `CARD`, `CORE`, `INTERNAL`, `DIGITAL` enum болгох |
| Үнэлгээ/төг | Дутуу | Security score байгаа ч энэ нь өөр ойлголт | `valuation_mnt` эсвэл `estimated_cost_mnt` талбар нэмэх |
| Тайлбар | Байгаа | `description` | Шууд ашиглана |
| Холбоотой системүүд | Хагас | Dependency map дараагийн шатанд гэж туссан | MVP-д `service_relations` many-to-many table нэмэх |
| Хөгжүүлэгч | Хагас | Owner team байгаа | `developer_name` эсвэл `developer_team` талбар нэмэх |
| Хугацаа | Хагас | `created_at`, `last_deploy_at` байгаа | `start_date`, `end_date` эсвэл `planned_duration` нэмэх |
| Ашиглагдаж байгаа эсэх | Хагас | `status` байгаа | `in_use` boolean эсвэл status enum-д `ACTIVE/INACTIVE` тодорхойлох |
| Мэдээллийн аюулгүй байдлын стандарт шалгах | Байгаа | Security controls, score, OWASP mapping байгаа | Даалгаварт ойлгомжтой checklist UI хийх |
| Жагсаалтаар харах | Байгаа | Services table гэж туссан | Хэрэгжүүлэх |
| Хайлт хийх | Байгаа | Filter/search гэж туссан | Name/type/developer/status хайлт хийх |

## 4. Одоогийн судалгаанд нэмэх шаардлагатай data model

Даалгаварт яг тааруулахын тулд `services` table-ийг дараах байдлаар шинэчлэх нь зөв.

```sql
services
- id
- service_key
- name
- type                 -- CARD, CORE, INTERNAL, DIGITAL
- valuation_mnt        -- PDF дээрх "Үнэлгээ/төг"
- description
- developer_name
- developer_team
- start_date
- end_date
- in_use               -- ашиглагдаж байгаа эсэх
- base_url
- health_url
- swagger_url
- repo_url
- environment
- status
- owner_team_id
- created_at
- updated_at
```

Холбоотой системүүд:

```sql
service_relations
- id
- source_service_id
- target_service_id
- relation_type        -- DEPENDS_ON, CALLS, INTEGRATES_WITH
- description
```

Security checklist:

```sql
security_standard_checks
- id
- service_id
- standard_name
- check_name
- result               -- PASS, FAIL, WARNING, NOT_CHECKED
- evidence
- checked_at
- checked_by
```

## 5. Хөгжүүлэлт хийхэд санал болгох MVP scope

Анхны даалгаврыг 100% биелүүлэх MVP:

1. Login/register эсвэл seed admin user.
2. Password hash хадгалах.
3. System registration CRUD.
4. PDF-ийн заавал талбаруудтай form:
   - нэр,
   - төрөл,
   - үнэлгээ/төг,
   - тайлбар,
   - холбоотой системүүд,
   - хөгжүүлэгч,
   - хугацаа,
   - ашиглагдаж байгаа эсэх.
5. System list table.
6. Search/filter:
   - нэрээр,
   - төрлөөр,
   - хөгжүүлэгчээр,
   - ашиглагдаж байгаа эсэхээр.
7. Security standard checklist:
   - HTTPS ашигладаг эсэх,
   - authentication байгаа эсэх,
   - role/permission байгаа эсэх,
   - audit log байгаа эсэх,
   - secret ил хадгалаагүй эсэх,
   - endpoint хамгаалагдсан эсэх.
8. Security score эсвэл pass/fail summary.

Enterprise нэмэлтээр харуулах MVP:

1. `Banking Transfer Service`-ийг registered system болгож бүртгэх.
2. Энэ service дээр health URL, swagger URL, security checklist бөглөх.
3. Дараа нь 2 хэрэглэгчийн харилцах данс хооронд transfer хийх demo харуулах.

## 6. Юуг түр хойшлуулах вэ

Дараах зүйлсийг эхний хамгаалалтад заавал хийх шаардлагагүй:

- Eureka Discovery Server.
- Config Server.
- Gateway dynamic route update.
- Олон тусдаа microservice.
- Full notification integration.
- Prometheus/Grafana monitoring.
- Complex fraud detection.

Эдгээрийг “дараагийн шатны enterprise өргөтгөл” гэж тайлбарлахад хангалттай.

## 7. Хөгжүүлэлт хийхэд хэр боломжтой вэ

Хөгжүүлэлт хийхэд **боломжтой, сайн чиглэлтэй**, гэхдээ scope-оо зөв барих хэрэгтэй.

Хамгийн зөв эхлэл:

```text
Phase 1: Даалгаврын үндсэн систем
React + Spring Boot + PostgreSQL
Login + system CRUD + list/search + security checklist

Phase 2: Enterprise харагдуулах нэмэлт
Health URL, Swagger URL, security score, audit log

Phase 3: Банкны бодит demo service
Banking Transfer Service буюу 2 данс хооронд гүйлгээ

Phase 4: Microservice architecture өргөтгөл
Gateway, Eureka, Config Server
```

Ингэж явбал анхны даалгаврын шаардлага алдагдахгүй, харин судалгаанд гаргасан enterprise architecture нь давуу тал болж харагдана.

## 8. Эцсийн санал

Одоогийн судалгааг шууд хөгжүүлэлт эхлэх суурь болгон ашиглаж болно. Гэхдээ хөгжүүлэлтийн хамгийн эхний backlog-ийг PDF даалгаврын exact шаардлага дээр түгжих хэрэгтэй.

Санал болгож буй төслийн богино нэр:

**System Registry & Security Compliance Platform**

Санал болгож буй хамгаалалтын тайлбар:

> Анхны даалгавар нь системийн бүртгэл хийх CRUD шаардлага байсан. Би үүнийг банкны байгууллагад илүү ойр Enterprise Service Catalog болгон өргөтгөж, систем бүрийн эзэмшигч, төрөл, холбоотой систем, ашиглалтын төлөв, аюулгүй байдлын checklist, health/API мэдээллийг нэг дор хянах платформ болгон хөгжүүлсэн.
