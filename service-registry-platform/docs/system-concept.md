## 1. Нэг өгүүлбэрээр

**Байгууллагын бүх системийг бүртгэж, аюулгүй байдлын стандартаар үнэлж, хяналтад байлгадаг төв платформ — бөгөөд «бүртгэгдсэн систем» гэдэг нь зөвхөн каталогийн мөр биш, бодитоор ажиллаж буй сервис байж болохыг банкны шилжүүлгийн жинхэнэ сервисээр нотолсон.**

## 2. Гол санаа: даалгавраас платформ руу

Анхны даалгавар нь «Системийн бүртгэл хийх» гэсэн CRUD хэлбэрийн шаардлага байв (нэвтрэх, систем бүртгэх, жагсаалт/хайлт, аюулгүй байдлын стандарт шалгах). Үүнийг гурван давхаргаар өргөтгөж концепц болгосон:

1. **Бүртгэл → Каталог.** Системийг нэр төдийгөөр биш — төрөл (Карт/Коре/Дотоод/Дижитал), үнэлгээ (₮), хөгжүүлэгч, хугацаа, ашиглалтын төлөв, хоорондын хамаарал (`DEPENDS_ON`/`CALLS`/`INTEGRATES_WITH`), base/health/swagger/repo URL-тай нь бүртгэдэг Enterprise Service Catalog болгосон.
2. **Шалгах хайрцаг → Compliance хэмжүүр.** «Стандарт шалгах» шаардлагыг OWASP (ASVS, API Security) суурьтай, жинлэсэн 8 control бүхий checklist болгож, систем бүрд PASS/WARNING/FAIL үр дүн, нотолгоо, шалгагч, огноог бүртгээд **Security Score (0–100)** тооцдог болгосон. WARNING нь жингийн 50%-иар тооцогддог тул «хагас биелэлт» ч хэмжигдэнэ.
3. **Каталогийн мөр → Ажиллаж буй систем.** Каталогт бүртгэгдсэн «Banking Transfer Service» нь бодит, тусдаа Spring Boot микросервис: өөрийн өгөгдлийн сан, Swagger, health endpoint, аудиттай. Ингэснээр «системийг бүртгэх → health/API-г нь холбох → аюулгүй байдлыг нь үнэлэх» гэдэг платформын бүрэн мөчлөг бодит жишээгээр харагдана.

## 4. Концепцийн гол зарчмууд

### 4.1. Нэг нэвтрэлт, хоёр систем (SSO)

Token олгогч нэг л газар — platform-api. Banking сервис ижил `JWT_SECRET`-ээр token-ыг **зөвхөн шалгадаг** resource server (өөрөө login-гүй). Хэрэглэгч порталд ч, банкны апп-д ч нэг эрхээрээ нэвтэрнэ. Энэ нь байгууллагын төв auth + домэйн сервисүүд гэсэн бодит загварыг дуурайсан шийдэл.

### 4.2. Сервис бүр өөрийн өгөгдөлтэй (database-per-service)

Хоёр сервис хоёр тусдаа PostgreSQL ашиглана; хооронд нь DB түвшний FK байхгүй. Банкны харилцагчийг платформын хэрэглэгчтэй `customers.username` ↔ JWT subject гэсэн **сул холболтоор** холбосон — cross-DB хамаарал үүсгэлгүйгээр эзэмшлийн хяналт хийдэг.

### 4.3. Эрх нь дүрээр, эзэмшил нь өгөгдлөөр

- Цэвэр дүрийн хаалт: `@PreAuthorize` (портал дээр ADMIN бичих эрхтэй, SECURITY checklist засна; банкин дээр ADMIN = теллер).
- Өгөгдлөөс хамаарах хяналт: ADMIN биш хэрэглэгч банкин дээр **зөвхөн өөрийн** данс/гүйлгээг харж, зөвхөн өөрийн данснаас шилжүүлнэ (`FORBIDDEN_ACCOUNT` 403).

### 4.4. Аюулгүй байдал — шалгадаг зүйл, мөн мөрддөг зарчим

Платформ бусад системээс шаарддаг стандартаа өөрөө мөрддөг: BCrypt нууц үг, stateless JWT, RBAC, нэвтрэлтийн түгжээ (5 алдаа → 15 мин, HTTP 423), CORS allowlist, бүх чухал үйлдлийн аудит, нууц утгын fail-fast (env байхгүй бол сервис асахгүй), demo seed-ийг prod-оос тусгаарласан Flyway байршил. Өөрөөр хэлбэл checklist-ийн мөр бүр кодод жишээтэй.

### 4.5. Банкны домэйныг «жинхэнэ банк шиг» загварчилсан

| Зарчим | Хэрэгжилт |
| --- | --- |
| Мөнгө алга болохгүй | Гүйлгээ бүр DEBIT+CREDIT давхар бичилт (ledger); дансуудыг эрэмбэтэй pessimistic lock-оор түгжинэ (deadlock-гүй, зэрэгцээ гүйлгээнд overdraw-гүй) |
| Гүйлгээ давхардахгүй | `Idempotency-Key`: давтан илгээхэд анхны гүйлгээ буцна, дахин гүйцэтгэхгүй |
| Татгалзал ч мөр үлдээнэ | Бизнес дүрмээр татгалзсан гүйлгээ FAILED төлөвтэй, шалтгаантайгаа хадгалагдана (үлдэгдэл хөндөгдөхгүй) — аудитын шаардлага |
| Алдаа засах зам нь буцаалт | Устгал/засвар байхгүй; зөвхөн теллер SUCCESS гүйлгээг нэг удаа REVERSED болгоно |
| Лимит | Нэг удаагийн (5 сая ₮) ба өдрийн зарлагын (10 сая ₮) тохируулгатай лимит; буцаалт лимитийг тэглэдэггүй |
| Мөр бүр хариуцлагатай | Банкны бүх үйлдэл actor snapshot-той аудит; алдаа гарсан ч аудит хадгалагдана (REQUIRES_NEW) |
| Машинд ойлгомжтой алдаа | 17 утгатай `ErrorCode` — UI үүгээр монгол мессеж харуулна |

### 4.6. Ажиллагааны сахилга

Схемийг зөвхөн Flyway өөрчилнө (`ddl-auto=validate`), тохиргоо env-first, 78 автомат тест хоёр сервисийг бүрхэнэ, GitHub Actions CI бэлэн, бүх стек Docker-оор нэг командаар асна.

## 5. Гол хэрэглээний урсгалууд (демо сценари)

1. **Админ (портал):** нэвтэрнэ → шинэ систем бүртгэнэ (бүх талбар + холбоотой системүүд) → Security checklist бөглөнө → score шууд тооцогдоно → Dashboard дээр нийт систем/дундаж score → Аудит логоос хэн юу хийснийг харна.
2. **Харилцагч (банкны апп, `batbold`):** нэвтэрнэ → зөвхөн өөрийн данс харагдана → шилжүүлэг хийнэ (ижил данс/лимит/үлдэгдлийн алдаа монголоор) → хуулгаас opening/closing үлдэгдэлтэй хөдөлгөөнөө харна. Бусдын данс руу хандвал 403.
3. **Теллер (банкны апп, `admin`):** харилцагч үүсгэнэ (CUST-дугаар автомат) → данс нээж/блоклож/хаана → буруу гүйлгээг буцаана (үлдэгдэл сэргэнэ, давхар буцаалт 409) → банкны аудит логийг харна.

## 6. Энгийн CRUD-ээс юугаараа ялгарах вэ

- Даалгаврын бүх шаардлага (нэвтрэлт, 8 талбартай бүртгэл, стандарт шалгалт, жагсаалт/хайлт) **100% биелсэн** дээр нь платформ болж өргөжсөн.
- Аюулгүй байдал нь «чеклист» биш — тоон үнэлгээ (score) + платформ өөрөө үлгэрлэн мөрддөг.
- Микросервис архитектурын гол ойлголтууд (төв auth, DB-per-service, resource server, каталог, health/Swagger холболт, аудит) нэг төсөлд бодитоор харагдана.
- Банкны домэйн нь сургалтын жишээ биш түвшинд: давхар бичилт, идемпотентлог, төлөвийн lifecycle, лимит, эзэмшил, буцаалт — бүгд зэрэгцээ ажиллагаанд тэсвэртэйгээр шийдэгдэж, тестээр баталгаажсан.

## 7. Төлөв ба хил хязгаар

**Хийгдсэн:** 2 backend (Spring Boot 3.5, Java 17), 2 frontend (React 19 + Vite), 2 PostgreSQL, Flyway (platform V1–V5, banking V1–V7), 78 автомат тест бүгд ногоон, E2E/зэрэгцээ ажиллагааны шалгалт жинхэнэ Postgres дээр хийгдсэн, бүрэн Docker стек.

**Зориуд хойшлуулсан (дараагийн шатны өргөтгөл):** Spring Cloud Gateway, Eureka Discovery, Config Server, notification (Email/Slack), Prometheus/Grafana. Эдгээр нь концепцид туссан ч эхний хүрээнд оруулахгүй байх шийдвэрийг scope-ийн үнэлгээгээр гаргасан (`төслийн concept/даалгаврын нийцлийн үнэлгээ.md` §6).

**Нээлттэй ажил:** GitHub push + CI идэвхжүүлэлт, хоёр апп-ын Vitest тестүүд, давхардсан frontend кодын shared package, порталын user удирдлагын API. Мөн хяналтын функцийг бодит болгох (health check, runtime status, poller) — шинэ хадгаламжийн сервис нэмэгдэхийн өмнөх бэлтгэл, `docs/registry-monitoring-research.md` харах.
